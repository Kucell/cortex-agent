# Log Legibility

> 对应任务：T-H18
> 状态：已实现第一版基线
> 日期：2026-04-21

---

## 1. 目标

为 Cortex Agent 提供一套最小但稳定的日志理解框架，让 Agent 在服务型项目中优先用日志确认“系统实际发生了什么”，而不是只靠代码猜测。

这份文档解决三个问题：

- 先看哪些日志字段
- 按什么顺序排查日志
- 如何把日志证据整理成稳定输出

---

## 2. 适用范围

适用于以下场景：

- 服务启动失败
- 接口返回异常
- 请求链路中断
- 线上问题回放到本地或预发环境

当前不限制具体技术栈，也不绑定某个日志平台。

---

## 3. 日志查看入口

Agent 在开始排查时，应优先确认“日志从哪里来”，而不是先陷入代码细节。

推荐入口顺序：

1. **当前终端输出**
   - 适合本地开发、服务启动、即时错误
2. **应用容器日志**
   - 适合 Docker / Compose / 容器化服务
3. **应用本地日志文件**
   - 适合有持久化日志目录的项目
4. **聚合日志平台**
   - 适合团队环境或多服务环境

结论：

- 先选离当前问题最近的日志源
- 如果已经在终端复现，不要先跳到远端平台
- 如果是多服务问题，不要只看单个进程日志

---

## 4. 优先关注的日志字段

无论日志格式如何，Agent 都应优先抽取这些字段：

| 字段 | 作用 | 说明 |
|------|------|------|
| `timestamp` | 时间定位 | 确认问题发生窗口 |
| `level` | 严重级别 | 区分 info / warn / error / fatal |
| `service` / `module` | 所属服务或模块 | 判断问题落在哪一层 |
| `message` | 核心事件 | 提取“发生了什么” |
| `request_id` / `trace_id` | 请求关联 | 用于串联上下游 |
| `path` / `route` | 接口入口 | 判断具体请求命中点 |
| `status_code` | 响应结果 | 快速识别 4xx / 5xx / 成功但异常 |
| `duration_ms` | 时延 | 判断慢请求或超时 |
| `error_type` / `stack` | 错误归因 | 判断代码异常、依赖异常或配置异常 |

最小要求：

- 至少要拿到 `时间 + 级别 + 模块/服务 + message`
- 如果是接口问题，尽量补齐 `request_id/path/status_code`

---

## 5. 标准排查顺序

Agent 不应一上来就全文扫描所有日志。建议固定为四步：

### Step 1：先定时间窗口

先确认：

- 问题大概什么时候发生
- 是持续复现还是偶发
- 是启动阶段还是运行阶段

如果时间窗口不明确，日志搜索范围会快速失控。

### Step 2：先找错误级别，再补上下文

优先顺序：

1. `fatal`
2. `error`
3. `warn`
4. `info`

但不能只截取单行错误，必须补前后上下文，至少确认：

- 错误前一个关键动作
- 错误对应的请求或模块
- 错误后是否有恢复或重试

### Step 3：先定位入口，再判断影响面

接口类问题优先问三个问题：

1. 请求是否进入系统
2. 请求进入后停在哪一层
3. 错误是单点失败还是链路级失败

这一步要尽量基于 `path`、`request_id`、`trace_id`、`service/module` 来回答。

### Step 4：最后再回到代码

只有当日志证据已经说明：

- 哪个模块出问题
- 哪类请求出问题
- 哪个阶段出问题

才进入代码定位。

原则是：

> 先用日志缩小问题面，再用代码确认根因。

---

## 6. 三类常见场景模板

### 6.1 启动失败

优先看：

- 进程启动顺序
- 配置加载结果
- 依赖连接状态
- 第一条 `error/fatal`

重点回答：

- 服务有没有真正启动完成
- 失败发生在配置、依赖连接、迁移、初始化还是监听端口阶段

### 6.2 接口失败

优先看：

- 请求是否进入路由
- 请求参数是否被接收
- 上游调用是否返回
- 响应码和错误栈

重点回答：

- 是请求没进系统，还是进了系统后失败
- 是业务错误、依赖错误还是权限/参数错误

### 6.3 链路中断

优先看：

- `request_id` / `trace_id`
- 上下游服务的对应日志
- 超时、重试、熔断、连接失败等信号

重点回答：

- 请求停在第几跳
- 是上游未发出、下游未收到，还是下游报错未回传

## 7. 推荐输出格式

Agent 在完成日志排查后，建议统一输出：

```text
## Log Findings

- 时间窗口：2026-04-21 10:12 ~ 10:14
- 问题入口：POST /api/login
- 影响模块：auth-service / login handler
- 关键 request_id：req-123

### Observations
- 请求已进入网关并转发到 auth-service
- auth-service 收到请求后在 token exchange 阶段报错
- 下游依赖返回 500，随后出现 2 次重试

### Conclusion
- 问题不在前端发起层，也不在网关路由层
- 当前更可能是 auth-service 与下游身份服务之间的依赖异常

### Next Step
- 继续检查 token exchange 对应调用点与下游服务日志
```

要求：

- 必须引用具体日志证据，而不是只给结论
- 必须指出影响层级
- 必须给出下一步动作

---

## 8. 使用边界

这份基线只定义“如何读日志”，不等于：

- 自动接入日志平台
- 自动解析所有日志格式
- 自动完成根因分析

它的作用是先把最小排查方法统一下来，为后续：

- `metrics-legibility`
- `trace-legibility`
- runtime evidence 接入 `/briefing` / `/ship`

提供稳定基础。

---

## 9. 当前结论

T-H18 的目标不是把日志系统做复杂，而是建立一个简单但稳定的共识：

> 在服务型问题里，Agent 应先看日志，再下结论；先缩小问题面，再回到代码。
