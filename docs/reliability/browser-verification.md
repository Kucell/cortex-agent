# Browser Verification

> 对应任务：T-H19
> 状态：已实现第一版基线
> 日期：2026-04-21

---

## 1. 目标

为 Cortex Agent 提供一套最小但稳定的浏览器验证基线，让 Agent 在 UI 项目中用真实页面交互确认“界面真的可用”，而不是只靠组件代码和静态推断。

这份文档解决四个问题：

- 先验证哪类页面和路径
- 最小交互动作是什么
- 截图和断言怎么留证
- 如何整理为可复用的验证结论

---

## 2. 适用范围

适用于以下场景：

- 页面是否能打开
- 按钮、链接和导航是否工作
- 表单是否能提交
- 关键 UI 流程是否走通
- 改动后是否出现明显视觉或交互回归

当前不绑定具体前端框架，也不强制绑定某个浏览器工具实现。

---

## 3. 浏览器验证入口

Agent 在开始 UI 验证时，应先确定验证对象，而不是直接随机点击页面。

推荐入口顺序：

1. **关键页面是否可打开**
   - 首页、登录页、目标功能页
2. **关键交互是否可触发**
   - 按钮点击、导航跳转、展开/关闭、切换状态
3. **关键输入是否可提交**
   - 表单、搜索框、筛选器、登录流程
4. **关键结果是否可观察**
   - 成功提示、跳转结果、列表变化、错误提示、截图

结论：

- 先确认“页面能打开”，再验证复杂交互
- 先做用户主路径，再做边缘路径
- 如果页面都打不开，不要继续做深层 UI 断言

---

## 4. 最小验证动作

对 UI 项目，Agent 至少应覆盖以下四类动作：

| 动作 | 目的 | 最小要求 |
|------|------|----------|
| 页面打开 | 确认入口可用 | 页面成功加载，无明显白屏/崩溃 |
| 交互触发 | 确认控件工作 | 按钮、链接、切换器能响应 |
| 输入提交 | 确认主流程可执行 | 表单或关键输入可提交 |
| 结果留证 | 确认结论可回看 | 截图、页面状态或断言结果可保存 |

对大多数 UI 变更，至少要回答：

- 页面是否打开成功
- 目标交互是否有响应
- 关键流程是否继续向前推进
- 是否留下了截图或断言证据

---

## 5. 标准验证顺序

Agent 不应把浏览器验证做成无目标的页面漫游。建议固定为五步：

### Step 1：先定目标路径

先明确：

- 要验证哪一个页面
- 这次改动影响哪条用户路径
- 成功标准是什么

如果没有目标路径，浏览器验证很容易变成表面点击。

### Step 2：先确认页面加载

优先检查：

- 页面是否成功打开
- 是否出现明显报错
- 是否有关键容器或主标题渲染完成

如果这一步失败，应先停下来确认：

- 路由问题
- 权限问题
- 服务未启动
- 首屏渲染异常

### Step 3：再做关键交互

优先顺序：

1. 点击主按钮或入口链接
2. 执行主路径导航
3. 填写关键输入
4. 触发表单提交或核心动作

这一步关注的是“交互是否被接收”，不是一次性把全部页面元素都测完。

### Step 4：检查页面反馈

至少确认以下一项：

- 页面跳转成功
- 成功提示出现
- 列表或状态发生变化
- 错误提示与预期一致

如果没有任何反馈，不能直接判定验证通过。

### Step 5：最后留证

建议至少保留以下一种证据：

- 最终页面截图
- 关键中间态截图
- 明确的页面断言结果

原则是：

> 浏览器验证不是“我点过了”，而是“我能证明点过之后发生了什么”。

## 6. 四类常见场景模板

### 6.1 页面打开验证

优先看：

- 路由是否可达
- 页面骨架是否渲染
- 是否有首屏错误提示

重点回答：

- 页面是否真正打开
- 首屏是否可用

### 6.2 按钮 / 链接交互验证

优先看：

- 点击后是否有响应
- 是否触发跳转、弹窗、展开或状态切换
- 是否出现静默无反应

重点回答：

- 控件是否可操作
- 交互结果是否符合预期

### 6.3 表单提交流程验证

优先看：

- 输入是否可填写
- 提交后是否出现 loading / 成功 / 错误反馈
- 是否有跳转或状态变更

重点回答：

- 表单是否能真正提交
- 失败时是否有可理解反馈

### 6.4 回归截图验证

优先看：

- 改动区域是否已渲染
- 布局是否明显错位
- 关键元素是否消失或重叠

重点回答：

- 本次变更是否引入明显可见回归
- 是否需要继续做更细粒度视觉比对

## 7. 推荐输出格式

Agent 在完成浏览器验证后，建议统一输出：

```text
## Browser Verification

- 目标页面：/login
- 目标流程：登录主路径
- 验证环境：本地开发环境
- 结果：通过

### Steps Executed
- 打开登录页
- 输入账号与密码
- 点击“登录”
- 等待跳转到控制台首页

### Evidence
- 截图：output/playwright/login-success.png
- 页面断言：首页主标题可见

### Conclusion
- 登录主路径已走通
- 当前未见明显交互阻断

### Next Step
- 如需更高信心，可继续补失败态与权限态验证
```

要求：

- 必须写明目标页面和目标流程
- 必须给出执行过的关键步骤
- 必须留下截图或断言证据
- 必须说明通过/失败/部分通过

---

## 8. 使用边界

这份基线只定义“如何做最小浏览器验证”，不等于：

- 自动覆盖所有页面
- 自动建立完整 E2E 测试
- 自动完成视觉回归系统
- 自动处理复杂登录或验证码

它的作用是先建立一个稳定共识，为后续：

- browser skill / workflow
- runtime evidence 接入 `/briefing` / `/ship`
- verification templates 标准化

提供基础。

---

## 9. 当前结论

T-H19 的目标不是把 UI 自动化做重，而是建立一个简单但稳定的共识：

> 在 UI 问题里，Agent 应先用真实页面验证主路径是否可用，并留下截图或断言证据。
