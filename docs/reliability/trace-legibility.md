# Trace Legibility

> 对应任务：T-H21
> 状态：已实现第一版基线
> 日期：2026-04-22

---

## 1. 目标

为 Cortex Agent 提供一套最小但稳定的链路理解框架，让 Agent 在多服务或多跳请求场景下，能基于 `request_id`、`trace_id` 或等价关联标识判断“请求走到了哪里、卡在了哪里、是哪一跳出了问题”。

这份文档解决四个问题：

- 链路排查从哪里开始
- `request_id / trace_id` 应该怎么用
- 如何判断请求断在哪一跳
- 如何把链路证据整理成稳定输出

---

## 2. 适用范围

适用于以下场景：

- 网关到服务的多跳请求排查
- 上下游依赖调用失败
- 超时、重试、熔断、重定向等链路级问题
- “日志看起来正常，但请求还是没走通”的场景

当前不绑定具体 tracing 平台，也不要求必须有完整分布式追踪系统。

---

## 3. 关联标识优先级

Agent 在排查链路问题时，应优先寻找能把同一次请求串起来的标识。

推荐优先级：

1. `trace_id`
2. `request_id`
3. `span_id`
4. 退化方案：`timestamp + path + user/session + module`

原则：

- 优先使用跨服务稳定传递的 ID
- 如果没有统一 trace_id，至少要在相邻服务间找到同一 request_id
- 如果完全没有关联 ID，必须显式说明链路结论的置信度较低

---

## 4. 链路查看入口

Agent 不应一上来就同时扫所有服务。推荐入口顺序：

1. **入口层**
   - 网关、BFF、API Gateway、Ingress
2. **主服务层**
   - 真正承接业务的核心服务
3. **下游依赖层**
   - 数据库、缓存、消息队列、外部 API、内部下游服务
4. **异常放大层**
   - 重试器、熔断器、超时处理器、补偿逻辑

结论：

- 先确认请求是否进入系统
- 再确认请求是否进入核心服务
- 最后确认问题是“没发出”“没收到”还是“收到了但执行失败”

---

## 5. 标准排查顺序

Agent 不应把链路排查做成无边界的跨服务全文检索。建议固定为五步：

### Step 1：先定入口请求

先明确：

- 哪个接口或动作触发问题
- 请求发生的大致时间窗口
- 是否能拿到 `trace_id / request_id`

如果入口请求都不明确，后续链路判断会失去锚点。

### Step 2：确认入口层是否接收请求

优先检查：

- 网关或入口服务是否收到请求
- 请求是否被转发
- 返回码和时延如何

重点回答：

- 请求有没有真正进入系统
- 是入口层就失败，还是后续某一跳失败

### Step 3：确认主服务是否接收并执行

优先检查：

- 是否存在同一 `trace_id / request_id`
- 主服务是否产生日志或 span
- 主服务在哪个阶段报错、超时或中断

重点回答：

- 请求是否到达主服务
- 主服务是没处理、处理中断，还是处理完但回传失败

### Step 4：确认下游依赖是否成功返回

优先检查：

- 下游调用是否发出
- 下游是否收到
- 下游返回了什么
- 是否触发超时、重试、熔断

重点回答：

- 问题是“上游没发出”“下游没收到”还是“下游报错”

### Step 5：最后判断断点位置

将结论压缩为以下几类之一：

- 断在入口层
- 断在主服务内部
- 断在下游依赖调用
- 断在返回路径
- 暂无法确定，需要补链路标识或更细日志

原则：

> Trace 的价值不是把所有 span 都看一遍，而是快速判断请求断在哪一跳。

---

## 6. 常见链路信号

Agent 在链路排查中，优先关注这些信号：

| 信号 | 说明 | 常见含义 |
|------|------|----------|
| `trace_id` / `request_id` 丢失 | 上下游无法串联 | 传播链断裂或日志不完整 |
| `timeout` | 请求耗尽等待时间 | 下游慢、网络堵塞、资源瓶颈 |
| `retry` | 自动重试触发 | 下游不稳定、系统在勉强维持 |
| `circuit_open` / `fallback` | 熔断或降级 | 下游异常已被放大到治理层 |
| `upstream 5xx` | 下游错误上抛 | 真正问题可能在依赖层 |
| `gateway 4xx/5xx` | 入口层直接失败 | 路由、权限、参数或网关层异常 |

---

## 7. 三类常见场景模板

### 7.1 请求未进入主服务

优先看：

- 网关日志
- 路由命中情况
- 鉴权或参数校验结果

重点回答：

- 请求是被入口拦住，还是转发前就失败

### 7.2 请求进入主服务但下游失败

优先看：

- 主服务中同一 `request_id / trace_id`
- 下游调用日志
- 超时、重试、熔断信号

重点回答：

- 是下游不可用，还是主服务调用逻辑异常

### 7.3 请求执行成功但返回链路异常

优先看：

- 主服务内部成功日志
- 回包日志
- 网关响应日志

重点回答：

- 是服务执行成功但响应未正确返回
- 还是返回后被网关、代理或客户端侧中断

## 8. 推荐输出格式

Agent 在完成链路排查后，建议统一输出：

```text
## Trace Findings

- 时间窗口：2026-04-22 14:05 ~ 14:08
- 问题入口：POST /api/login
- 关键 trace_id：trace-abc
- 断点判断：下游身份服务调用阶段

### Hops
- Gateway：收到并转发成功
- Auth Service：收到请求，发起 token exchange
- Identity Service：返回 500
- Response Path：未正常回到前端

### Conclusion
- 请求已进入系统，也进入主服务
- 当前更可能断在 auth-service 调用 identity-service 这一跳

### Next Step
- 继续检查 identity-service 的错误日志与相关指标
```

要求：

- 必须写清入口请求
- 必须写清关键关联 ID
- 必须标出断点判断
- 必须给出下一步动作

---

## 9. 使用边界

这份基线只定义“如何读链路”，不等于：

- 自动接入 tracing 平台
- 自动生成完整 span 图谱
- 自动替代日志与指标排查
- 自动保证所有服务都携带 trace_id

它的作用是先建立一个稳定共识，为后续：

- runtime evidence 接入 `/briefing` / `/ship`
- verification templates 标准化

提供基础。

---

## 10. 当前结论

T-H21 的目标不是把 tracing 体系做重，而是建立一个简单但稳定的共识：

> 在多跳请求问题里，Agent 应先基于 request_id / trace_id 判断请求断在哪一跳，再决定往日志、指标还是代码继续下钻。
