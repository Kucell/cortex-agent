# Metrics Legibility

> 对应任务：T-H20
> 状态：已实现第一版基线
> 日期：2026-04-22

---

## 1. 目标

为 Cortex Agent 提供一套最小但稳定的指标理解框架，让 Agent 在运行时问题排查和交付验证中，能通过指标判断“系统是否健康、问题是否扩散、优化是否真的生效”。

这份文档解决四个问题：

- 应优先看哪些指标
- 指标应该如何分层
- 健康度如何表达
- 如何把指标证据整理成稳定输出

---

## 2. 适用范围

适用于以下场景：

- 发布后健康度确认
- 接口性能退化排查
- 错误率上升或波动排查
- 资源使用异常排查
- 业务关键流程是否稳定

当前不绑定具体监控平台，也不要求统一指标系统实现。

---

## 3. 指标分层

Agent 在看指标时，不应把所有图表平铺处理。建议固定分成三层：

### 3.1 系统指标

关注：

- CPU
- 内存
- 磁盘
- 网络
- 进程/容器重启

作用：

- 判断问题是否来自基础资源层

### 3.2 接口指标

关注：

- 请求量
- 错误率
- 延迟
- 超时
- 重试率

作用：

- 判断服务对外行为是否稳定

### 3.3 业务指标

关注：

- 登录成功率
- 下单成功率
- 支付完成率
- 搜索成功率
- 关键流程转化率

作用：

- 判断“系统看起来正常”是否真的等于“业务可用”

原则：

> 先看系统是否还能站住，再看接口是否稳定，最后看业务是否真正走通。

---

## 4. 优先关注的指标

如果没有完整指标体系，Agent 至少应优先找这几类：

| 指标 | 作用 | 常见信号 |
|------|------|----------|
| `error_rate` | 判断故障面 | 是否出现明显抬升 |
| `latency_p95/p99` | 判断体验退化 | 是否显著高于基线 |
| `throughput/qps` | 判断流量变化 | 是否突然下降或异常飙升 |
| `timeout_rate` | 判断依赖或链路阻塞 | 是否集中在某段时间抬升 |
| `retry_rate` | 判断系统是否在勉强维持 | 是否出现重试放大 |
| `cpu/memory` | 判断资源瓶颈 | 是否接近上限或持续抖动 |
| `success_rate` | 判断关键路径是否有效 | 是否看似成功但业务不通 |

最小要求：

- 至少拿到 `错误率 + 延迟 + 流量`
- 如果有业务关键路径，尽量再补 `success_rate`

---

## 5. 标准查看顺序

Agent 不应先盯某一张图表过度解读。建议固定为四步：

### Step 1：先定观察窗口

先确认：

- 问题从什么时候开始
- 是发布后出现，还是自然波动
- 要看 5 分钟、1 小时还是 24 小时窗口

如果窗口不清晰，指标很容易被噪声误导。

### Step 2：先看错误率和延迟

优先顺序：

1. `error_rate`
2. `latency_p95/p99`
3. `throughput`

原因：

- 错误率回答“是否坏了”
- 延迟回答“是否变慢了”
- 吞吐回答“是故障还是流量变化”

### Step 3：再分层判断问题落点

建议顺序：

1. 系统指标是否异常
2. 接口指标是否异常
3. 业务指标是否异常

不要只因 CPU 高就判断根因，也不要只因成功率高就判断业务无问题。

### Step 4：最后结合日志和链路

指标主要负责：

- 判定异常是否存在
- 判定影响范围
- 判定是否恢复

但指标通常不能单独给出根因，所以最后应回到：

- `log-legibility`
- `trace-legibility`

来缩小根因范围。

## 6. 健康度表达方式

如果需要把指标结果压缩成单次结论，建议统一分成三档：

### Healthy

条件示例：

- 错误率稳定在基线范围
- 延迟无明显抬升
- 关键业务成功率正常

输出建议：

- 可以继续推进，但保留观察

### Advisory

条件示例：

- 延迟轻微上升
- 错误率有波动但未扩散
- 资源利用率接近阈值

输出建议：

- 需要关注，建议继续结合日志和发布变更排查

### Attention

条件示例：

- 错误率明显抬升
- p95/p99 延迟显著劣化
- 关键业务成功率下降
- CPU/内存/重试率出现异常放大

输出建议：

- 需要优先排查，不能仅以“功能可点通”判定通过

原则：

> 指标健康度不是替代根因分析，而是帮助 Agent 先判断问题是否值得升级处理。

---

## 7. 三类常见场景模板

### 7.1 发布后确认

优先看：

- 发布前后错误率
- 发布前后延迟
- 关键业务成功率

重点回答：

- 本次发布是否引入健康度回退
- 是否需要继续观察还是立即回滚

### 7.2 性能退化排查

优先看：

- p95/p99 延迟
- 吞吐变化
- CPU / 内存 / GC 等资源指标

重点回答：

- 是流量变化导致，还是性能真的退化
- 是资源瓶颈还是接口层问题

### 7.3 业务流程异常

优先看：

- 关键成功率
- 错误率
- 上下游接口延迟和超时率

重点回答：

- 是技术层坏了，还是业务流程中断了
- 是否存在“接口成功但业务失败”的假健康状态

---

## 8. 推荐输出格式

Agent 在完成指标分析后，建议统一输出：

```text
## Metrics Findings

- 观察窗口：2026-04-22 09:00 ~ 10:00
- 健康度：Advisory
- 影响范围：login API / auth flow

### Observations
- error_rate 从 0.2% 升到 2.8%
- p95 latency 从 180ms 升到 740ms
- throughput 基本稳定，无明显流量飙升
- CPU 和 memory 无明显异常

### Conclusion
- 当前更像接口层或依赖层退化，不像系统资源瓶颈
- 问题集中在 login 相关路径

### Next Step
- 继续结合 auth-service 日志和 trace 判断慢点与错误点
```

要求：

- 必须写清观察窗口
- 必须指出健康度结论
- 必须说明影响层级
- 必须给出下一步动作

---

## 9. 使用边界

这份基线只定义“如何读指标”，不等于：

- 自动接入监控平台
- 自动建立统一指标面板
- 自动完成容量规划
- 自动替代日志和链路排查

它的作用是先建立一个稳定共识，为后续：

- `trace-legibility`
- runtime evidence 接入 `/briefing` / `/ship`
- verification templates 标准化

提供基础。

---

## 10. 当前结论

T-H20 的目标不是把监控体系做重，而是建立一个简单但稳定的共识：

> 在运行时问题里，Agent 应先用指标判断健康度和影响范围，再结合日志与链路继续下钻。
