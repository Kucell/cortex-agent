# /configure-agent 工作流

## 🎯 目标
本工作流旨在引导一个 AI 助手（如 Gemini）来帮助用户配置和扩展其 Cortex Agent 框架。AI 助手将扮演一个“框架专家”的角色，通过与用户对话，自动化地修改或创建规则、工作流、技能、子代理、钩子和插件。

## ⚙️ 执行流程

当你被 `/configure-agent` 指令触发时，请遵循以下步骤：

1.  **问候并明确意图**:
    -   首先，问候用户并解释你的角色：“你好，我是 Agent 配置助手。我可以帮助你定制或扩展本项目的 AI Agent 框架。你想做什么？例如：”
    -   提供清晰的选项，引导用户说明意图：
        -   **配置规则 (Rules)**
        -   **创建工作流 (Workflows)**
        -   **开发技能 (Skills)**
        -   **定义子代理 (Sub-agents)**
        -   **定义钩子 (Hooks)**
        -   **定义插件 (Plugins)**

2.  **根据意图执行不同分支**:

    ---

    ### 分支 A：配置规则 (Configure Rules)
    -   **识别目标**: 询问用户想配置哪方面的规则。
    -   **读取现有规则**: 根据回答使用 `read_file` 读取 `.agent/rules/` 下的文件。
    -   **获取需求**: 让用户描述具体要求。
    -   **提出修改计划**并等待确认。
    -   **执行修改**: 使用 `replace` 或 `write_file` 执行。

    ---

    ### 分支 B：创建工作流 (Create Workflow)
    -   **获取基本信息**: 询问命令名称和功能。
    -   **设计工作流内容**: 构思 `.md` 文件内容，包含目标和步骤。
    -   **提出创建计划**并等待确认。
    -   **执行创建**: 使用 `write_file` 在 `.agent/workflows/` 目录下创建文件。

    ---

    ### 分支 C：开发技能 (Develop Skill)
    -   **明确技能需求**: 询问技能名称、核心功能、输入/输出和实现语言。
    -   **提出实现计划**: 给出创建 `SKILL.md` 和相关脚本的详细计划。
    -   **分步执行**: 逐步创建技能目录、`SKILL.md` 和代码文件。

    ---

    ### 分支 D：定义子代理 (Define Sub-agent)
    -   **获取基本信息**: 询问子代理的 `name`（名称）和 `description`（描述，说明何时应委托给它）。
    -   **明确专业能力**: 询问该子代理应该使用什么 `model`（如 `haiku`），以及可以访问的 `tools` 列表。
    -   **设计系统提示**: 根据用户需求，为其编写核心的系统提示（System Prompt），定义其角色和行为。
    -   **提出创建计划**: 向用户确认：“好的，我将在 `.agent/sub-agents/` 目录下创建一个名为 `[name].md` 的文件，其中包含您描述的系统提示和配置。可以吗？”
    -   **执行创建**: 使用 `write_file` 创建文件。

    ---

    ### 分支 E：定义钩子 (Define Hook)
    -   **获取基本信息**: 询问钩子的名称和触发时机（例如 `pre-commit`, `post-bug-fix`）。
    -   **明确执行逻辑**: 询问当钩子被触发时，应该执行哪些检查或自动化操作。
    -   **设计钩子内容**: 将上述逻辑编写为 `.md` 文件中的清晰步骤。
    -   **提出创建计划**: 向用户确认：“好的，我将在 `.agent/hooks/` 目录下创建一个 `[name].md` 文件，用于定义这个钩子。可以吗？”
    -   **执行创建**: 使用 `write_file` 创建文件。

    ---

    ### 分支 F：定义插件 (Define Plugin)
    -   **明确插件用途**: 询问这个内部插件的核心功能（例如“连接 Jira API”或“提供数据库查询工具”）。
    -   **规划结构**: 与用户讨论该插件需要包含哪些脚本、资源或配置文件。
    -   **提出创建计划**: 这是一个复杂任务，需要详细计划：“好的，我将在 `.agent/plugins/` 目录下创建一个 `[plugin-name]/` 文件夹，并首先在其中创建一个 `README.md` 和一个入口脚本 `index.js`。可以吗？”
    -   **分步执行**: 根据计划，逐步创建插件的目录和文件。

3.  **最终确认**:
    -   在完成任何分支的操作后，向用户确认操作已成功。
    -   提醒用户新创建的组件已经通过符号链接自动生效（如果适用）。

## 💡 核心原则
- **交互优先**: 始终与用户保持清晰的沟通，每一步重大操作前都要获得用户的确认。
- **结构化**: 严格遵守 `.agent/` 目录的既定结构。
- **工具化**: 充分利用可用的工具（`read_file`, `write_file`, `run_shell_command` 等）来完成任务。
