## 技巧集合

### 🧪 TDD 工作流 (Test-Driven Development)

**红绿重构循环**：
1. **Red** — 先写一个会失败的测试，明确「完成」的定义
2. **Green** — 写最少量代码让测试通过（哪怕很丑）
3. **Refactor** — 在测试保护下安全重构

```
❌ 不要先写实现再补测试
✅ 测试描述行为，不描述实现细节
✅ 每次只红一个测试
```

**测试命名模板**：
```
[被测单元]_[场景条件]_[期望结果]
例：login_withExpiredToken_throwsAuthError
```

### 🐛 调试策略 (Debugging)

**橡皮鸭调试**：向自己（或 AI）逐行解释代码逻辑，往往在解释过程中自己找到问题。

**二分法定位**：
- 问题范围大时，在中间点加断点/日志
- 确认「上半段正常 / 下半段有问题」后，继续二分

**假设驱动调试**：
1. 提出假设：「问题可能出在 X」
2. 设计最小复现：移除所有无关代码
3. 验证或否定假设
4. 循环直到定位

**常见 Bug 类型速查**：
| 症状 | 优先检查 |
| :--- | :--- |
| 偶发性崩溃 | 竞态条件、未初始化变量 |
| 第一次正常，第二次失败 | 状态未清理、缓存污染 |
| 本地正常，生产失败 | 环境变量、时区、依赖版本 |
| 只有某些用户失败 | 权限、数据边界值、特殊字符 |

### 🔀 Git 技巧 (Git)

**提交粒度**：每个提交只做一件事，`git bisect` 时才能精准定位。

**常用救场命令**：
```bash
# 撤销最后一次提交（保留修改）
git reset --soft HEAD~1

# 暂存当前修改去修 bug
git stash && git stash pop

# 只提交部分修改
git add -p

# 查看某行代码的历史
git log -p -S "关键代码片段" -- path/to/file

# 找到引入 bug 的提交
git bisect start
git bisect bad HEAD
git bisect good v1.0.0
```

**分支命名规范**：
```
feat/T-001-user-auth
fix/T-023-login-crash
refactor/T-045-cleanup-utils
```

### ♻️ 重构技巧 (Refactoring)

**重构前提**：必须有测试覆盖，否则先补测试。

**小步重构原则**：
- 每次只改一个方面（命名、提取函数、移动模块）
- 每改一步就运行测试
- 不混合重构和功能变更

**提取函数时机**：
```
✅ 代码块超过 10 行
✅ 同样逻辑出现 2 次以上
✅ 需要注释才能理解的代码块
❌ 不要为了"整洁"过度提取
```

### 👀 代码审查视角 (Code Review)

**作为作者提交 PR 前自检**：
- [ ] 函数是否单一职责？
- [ ] 有无硬编码的魔法数字/字符串？
- [ ] 错误路径是否都有处理？
- [ ] 是否有不必要的注释（应删除）或缺少必要的注释？

**作为审查者**：
- 先看整体设计，再看细节
- 区分「必须改」和「建议改」
- 夸好的代码，不只挑问题

### ⚡ 性能优化时机

```
过早优化是万恶之源 — Donald Knuth

正确顺序：
1. 先让它工作（Make it work）
2. 再让它正确（Make it right）
3. 最后让它快（Make it fast）—— 只在需要时
```

**优化前必做**：用 profiler 找到真正的瓶颈，不要凭感觉猜。
