# GitHub 项目调研

使用 DeepWiki 做导航和跨文件定位；以 GitHub 与检出的源码为事实依据。

## 工作流

1. 将 `https://github.com/<owner>/<repo>` 规范化为 `https://deepwiki.com/<owner>/<repo>`。
2. 打开 DeepWiki 总览；若页面提供索引 commit 和生成时间，记录它们。
3. 从目录中选择与问题相关的架构、生命周期、扩展、安全、状态和测试页面。
4. 记录每项关键结论所引用的源文件与行号。
5. 在同一 commit 或当前 GitHub/本地源码中核验。若当前 HEAD 与索引版本不同，说明差异并以当前源码为准。
6. 在提出采纳建议前，检查 README、官方文档、许可证、安全说明和近期变更。
7. 对照目标项目约束，将每项思路标记为 `采纳`、`改造后采纳`、`延后` 或 `不采纳`。

若 DeepWiki 未收录、不可用或仓库为私有仓库，退化为 GitHub 代码搜索、本地浅克隆、官方文档和仓库原生导航工具。

## 证据规则

- 明确区分已核验事实、DeepWiki 摘要和自主推断。
- 安全、隐私、许可、API 兼容性和当前行为不得只以 DeepWiki 为证据。
- 优先引用固定到 commit 的永久链接，不只引用分支 HEAD。
- 未经明确授权，不得向 DeepWiki 提交或上传私有仓库内容。
- 同时记录负面证据：索引过期、文件缺失、文档冲突或行为无法核验。

## 输出契约

依次输出：简短结论、仓库与索引版本、相关模块地图、带一手证据的发现、面向目标项目的采纳分类表、风险，以及 DeepWiki 导航页和源码直链。

DeepWiki 链接用于导航；源码链接用于支撑结论。
