# 代码提交工作流 (/commit)

1. **状态检查**:
    - 运行 `git status` 查看待提交的改动。
2. **代码自检与评估**:
    - 对当前改动进行最后的自我审查，确保没有多余的调试代码或不符合规范的改动。
3. **质量验证**:
    - 运行项目配置的类型检查工具。
    - 运行相关的单元测试，确保无回归问题。
## 4. 生成提交信息
- **读取任务计划**: **必须读取** `.agent/plans/task-progress.md` 文件，找到当前 `in-progress` 的任务。
- **遵循提交规范**: 参考 `.agent/rules/commit-standards.md`。
- **生成结构化信息**: 
    - 确定改动的 `type` 和 `scope`。
    - 根据项目习惯编写 `subject`。
    - 在正文或脚注中**自动引用**相关任务 ID (例如 `[refs #123]`)。

## 5. 执行提交
- 确认生成的提交信息无误。
- 执行 `git commit -m "[提交信息]"`。

## 6. 后续操作
- 确认是否需要推送到远程仓库 (`git push`)。

---
## 🤖 高级用法：自动化集成

- **Pre-commit Hook**: 可以配置一个 Git `pre-commit` 钩子，在执行 `git commit` 时自动调用此 `/commit` 工作流，实现提交前的全自动化质量检查和信息生成。
- **Post-commit Hook**: 可以在此工作流的第 6 步之后，触发一个 `post-commit` 钩子，以执行例如自动 `git push` 或启动 CI/CD 流水线等操作。
