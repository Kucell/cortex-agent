# 代码审查工作流 (/code-review)

> **高级用法**: 可以将整个审查任务委托给一个专门的 `code-reviewer` 子代理来执行，以获得更专业的分析。

## 1. 准备阶段：上下文同步
- **读取所有规则**: 在开始审查前，**必须首先读取** `.agent/rules/` 目录下的所有规则文件（`architecture-design.md`, `code-standards.md` 等），以获得最全面的审查上下文。

## 2. 架构合规性审查
- 调用 `architecture-audit` 技能验证代码变动是否符合项目定义的层级结构。
- 检查职责划分是否合理，是否存在逻辑越层（Layer Violations）。
- 验证改动是否符合既定的设计模式或抽象准则。

## 3. 代码规范与质量
- **类型安全**: 检查是否合规使用类型系统，避免滥用弱类型。
- **命名一致性**: 确保变量、函数、类的命名清晰且符合项目风格。
- **文档完备性**: 检查核心 API 是否包含必要的注释（如 JSDoc/TSDoc）。
- **清洁度**: 移除多余的日志、注释代码或未运行的变量。

## 4. 测试完整性
- 检查新功能或修复是否伴随有测试用例。
- 运行测试套件，确认全量测试通过。

## 5. 性能与鲁棒性
- 使用 `code-evaluation` 技能识别潜在的性能隐患（如：高频循环中的 O(n^2) 操作）。
- 检查异步流程中的错误截获和资源清理。
- 评估边界条件的处理能力。

## 6. 文档同步
- 检查相关的 README 或技术文档是否根据代码改动进行了同步更新。
- 对于显著的逻辑变更，建议生成对应的 Mermaid 图示以增强可读性。
